💡 解法：在 git commit 之后立刻触发 AI 审查，把问题消灭在本地，不等 PR，不等同事。

先说为什么要做这个。代码审查这件事，我总结了三个核心痛点。

第一个，等待成本高。你提了 PR，同事可能在忙别的，等两天才给你 review，这两天你的思路早就断了，改起来很痛苦。

第二个，人工容易漏。空指针、SQL 注入、边界条件——这些问题在疲劳状态下很难每次都抓到。不是同事不认真，是人本来就不擅长这种机械性的扫描工作。

第三个，反馈周期长。问题在 PR 阶段才发现，这时候代码已经写完了，改起来成本很高，有时候还要推翻重来。

所以我的解法是——在 git commit 之后立刻触发 AI 审查，把问题消灭在本地，不等 PR，不等同事。接下来我讲怎么实现的。

来看架构设计。整个项目用 pnpm monorepo 组织，分成 5 个 package。

上面三个是核心：apps/api 是 Fastify 后端服务，提供 REST 接口和 SSE 实时推送；apps/vscode-extension 是 VS Code 插件，用 React 做侧边栏 UI；packages/review-engine 是审查引擎，负责 diff 过滤和并行调用 LLM。

下面两个是支撑层：llm-client 封装了三个 LLM 提供商，对外暴露统一接口；contracts 定义共享类型，storage 是内存存储层。

整个数据流很清晰——git commit 触发，review-engine 解析 diff，llm-client 调 AI，结果通过 SSE 推送到 VS Code 插件展示。每个环节职责单一，可以独立替换。

// part 03 · core modules

审查引擎 · 三步走

① filterDiff

过滤掉图片、lock 文件、CSS、自动生成代码——只把有意义的 diff 送给 LLM，节省 token。

② splitDiffByFile

按文件拆分 diff，并行调用 LLM，每个文件独立审查，互不干扰，速度快。

③ runReview

编排整个流程，支持进度回调，状态机：排队中 → 审查中 → 通过 / 有问题 / 失败。

// Prompt 工程：要求模型返回严格 JSON const prompt = `分析以下代码 diff，返回 JSON： { "summary": "...", "issues": [{ "severity": "error|warning|info", "file": "...", "line": N, "message": "..." }] }`

来看审查引擎的核心逻辑，分三步。

第一步，filterDiff。不是所有文件都需要 AI 看——图片、lock 文件、CSS、自动生成的代码，送给 LLM 是浪费 token。这一步把这些过滤掉，只保留有意义的代码变更。

第二步，splitDiffByFile。把 diff 按文件拆分，然后并行调用 LLM。为什么并行？因为一次 commit 可能改了 10 个文件，串行调用太慢，并行可以把延迟压到单个文件的审查时间。

第三步，runReview 编排整个流程，支持进度回调，这样前端可以实时显示"正在审查第 3/10 个文件"。

Prompt 工程这里有个关键设计——要求模型返回严格 JSON，包含 summary 和 issues 数组，每个 issue 有 severity、file、line、message。这样后端可以直接解析，不需要再做文本提取。

// part 04 · triggers

三种触发方式，覆盖全场景

🔧
① 手动触发 · POST /review/run

直接调 API，适合 CI 脚本或临时审查。传入 repo 名 + diff 内容，立即返回 run ID，通过 SSE 监听进度。

⚡
② git post-commit hook · 本地自动

每次 git commit 后，.githooks/post-commit 在后台异步调用 API，不阻塞提交，开发者无感知。

🚀
③ GitHub Actions · PR 自动审查

PR 打开或更新时，.github/workflows/pr-review.yml 自动触发，审查结果可以作为 PR check 展示。

触发机制是这个项目的一个亮点，我设计了三种方式，覆盖不同场景。

第一种，手动触发。直接调 POST /review/run 接口，适合 CI 脚本或者你想临时审查某段代码。接口是异步的，立即返回 run ID，然后通过 SSE 监听进度。

第二种，git post-commit hook。这是我最喜欢的方式。每次 git commit 之后，hook 脚本在后台异步调用 API，完全不阻塞提交，开发者感知不到。等你写下一行代码的时候，审查结果已经出来了，VS Code 侧边栏自动更新。

第三种，GitHub Actions。PR 打开或更新时自动触发，审查结果可以作为 PR check 展示，团队 review 之前就能看到 AI 的意见。

三种方式可以同时用，也可以只选一种，根据团队习惯来。

// part 04 · api

REST + SSE · 实时推送

# 触发审查 POST /review/run # 获取最新结果 GET /review/latest/:repo # SSE 实时进度流 GET /review/stream/:repo # 历史事件 GET /review/events/:repo # 用户反馈 POST /feedback

ReviewStatus 状态机

queued → reviewing → passed / has_issues / failed

问题严重级别

error warning info

用户反馈

API 设计这里有几个值得说的点。

接口设计很简洁，5 个端点覆盖所有场景。最关键的是 SSE 实时推送——GET /review/stream/:repo 是一个长连接，审查进度实时推送给客户端，VS Code 插件订阅这个流，所以你能看到"正在审查第 3 个文件"这样的实时状态。

状态机设计也很清晰：排队中 → 审查中 → 通过 / 有问题 / 失败。这个状态在 VS Code 侧边栏有对应的 UI 展示。

还有一个 /feedback 接口，让用户对每个 issue 给反馈——接受、忽略、还是需要更多上下文。这个数据可以用来持续优化 Prompt，让 AI 越来越准。

VS Code 插件是整个系统的"最后一公里"，也是开发者最直接感受到的部分。

侧边栏用 React 做的 Webview，展示审查概览和问题列表。支持按严重级别筛选——只看 error、只看 warning——还支持关键词搜索。通过 SSE 实时更新，不需要手动刷新。

更重要的是 VS Code Diagnostics 集成。问题直接标注在编辑器对应行，就像 ESLint 报错一样，红色波浪线，鼠标悬停看详情，点击跳转到问题位置。这个体验对开发者来说是最自然的。

还有一个我很喜欢的功能——复制修复提示词。点一下，自动生成一段结构化 Prompt，包含问题描述、文件路径、行号、上下文代码，直接粘贴给 AI 就能修复。这样从发现问题到修复，整个工作流都在 VS Code 里完成，不需要切换工具。
